# overview

## コンセプト

「開発者が困った瞬間に、3行で答えを出す」英語圏向けリファレンスサイト。

チートシートでも技術ブログでもなく、「特定の問題 → その答え」を1ページで完結させることを徹底する。

---

## ターゲット市場

- **主対象**: 英語圏の開発者（北米・欧州・東南アジア）
- **言語**: 英語のみで開始
- **想定ユーザー**: 実務で詰まって検索している開発者

日本語市場は対象外。英語圏ファーストにする理由は、市場規模・被リンク・検索ボリュームが桁違いであるため。

## コンテンツ方針

### 1ページ1問題1答え

各ページは「ひとつの問題に対してひとつの答えを返す」設計にする。

- 良い例: `git detached HEAD how to fix` → そのやり方だけを書く
- 悪い例: `git commands reference` → 全コマンドをリスト化するだけ

**汎用チートシートは作らない。**

### ロングテール検索意図を狙う

競合が強い短いキーワードは避け、「開発者が困ったときに検索する具体的なフレーズ」を狙う。

| 狙わない | 狙う |
| --- | --- |
| git commands | git reset soft mixed hard difference |
| docker tutorial | docker container keeps restarting fix |
| bash reference | bash check if file exists |
| regex tutorial | regex remove html tags javascript |

### エラー解決を中心に置く

「エラーメッセージ + 解決策」ページは検索意図が特に強い。Stack Overflow の回答は古くなりやすく、複数の回答が混在して読みにくい。「きれいに整理された1ページ」が存在しないことが多いため、そこを埋める。

---

## ページ構成テンプレート

各ページは以下の構成で統一する。

```
[H1] 問題を表すタイトル（検索クエリに近い形）

[Quick Answer]  ← 最重要セクション、ページ最上部に置く
  コードブロック + コピーボタン
  3〜5行で答えを完結させる

[When this happens]
  このエラーが起きる状況を1〜2文

[Explanation]（オプション、折りたたみ可）
  なぜそうなるかの説明

[Related]
  関連する問題・ページへの内部リンク（3〜5件）
```

「Quick Answer」を最上部に置くことで、検索ユーザーの満足度を高める（ページ内スクロールを最小化する）。

---

## 差別化ポイント

| 競合 | 弱点 | このサイトの優位性 |
| --- | --- | --- |
| Stack Overflow | 複数の回答が混在、古い情報が残る | 1答え、整理されている |
| 公式ドキュメント | 網羅的で長い、目的の答えまでが遠い | 問題直結で短い |
| GeeksForGeeks / W3Schools | SEO目的の薄い記事 | 実務で使えるコードに絞る |
| 汎用チートシート（Devhints 等） | 広すぎて検索意図と一致しない | 1問題1ページで一致率が高い |

## 内部リンク戦略

開発知識はグラフ構造（`git reset` → `git rebase` → `git reflog` のように連鎖する）なので、内部リンクを最初から設計する。

- 各ページの「Related」セクションに3〜5件の関連ページを置く
- カテゴリ索引ページ（`/git/`、`/docker/` 等）は各ページからのバックリンクを集める
- タグ軸（`#undo`、`#cleanup`、`#permission` 等）でも横断リンクを貼る

内部リンク密度が高いほど、個別ページの評価がサイト全体に波及しやすい。

---

## SEO 方針

- **タイトル**: 検索クエリに近い形で書く（「How to fix X」「X not working」「X vs Y difference」）
- **description**: Quick Answer セクションの冒頭1〜2文をそのまま使う
- **canonical**: 各ページに自己参照 canonical を設定する
- **sitemap.xml**: 全ページを含む形でビルド時に自動生成する

ドメインの選定も重要。「dev」「fix」「ref」「error」のいずれかを含めることで、ドメイン名自体が検索意図と一致しやすくなる。

## 成長フェーズ

### Phase 1: 最初の100ページ（`mvp-candidates.md` 参照）

- エラー解決系に集中する
- 各ページに Quick Answer + Related を必ず含める
- Google Search Console に登録して流入キーワードを監視する

### Phase 2: データで深掘る

- 「表示回数は多いがクリック率が低い」ページ → タイトル・description を改善
- 「想定外のクエリで流入している」ページ → そのクエリに向けてコンテンツを追記
- 「流入が多いカテゴリ」 → そのカテゴリのページを優先的に増やす

### Phase 3: ページ数を増やす

- Phase 2 の調査結果を元に、需要があるカテゴリを拡充
- 各カテゴリ50〜100ページを目標に積み上げる
- 内部リンク構造を見直し、カテゴリ索引ページの充実度を上げる

---

## やらないこと

- 汎用チートシート（「Git全コマンド一覧」など）
- 技術解説ブログ（長文のチュートリアル）
- 日本語コンテンツ（Phase 1 では）
- サーバーサイド処理（静的サイト縛り）
- アカウント・ログイン機能

## コンテンツ品質ガイドライン

以下のページは作らない。

- 「10 Best Git Commands」のような汎用まとめ
- 答えが「公式ドキュメントを読んでください」で終わるページ
- コードサンプルのないページ
- 動作確認なしにAIで量産したページ
